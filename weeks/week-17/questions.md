# Вопросы для самопроверки

### 1. Какие критерии вы использовали при разделении монолита на микросервисы?
При проектировании использовались следующие критерии:
* **Единая ответственность (Single Responsibility)**: Каждый сервис изолирует конкретную бизнес-область.
* **Независимость развертывания**: Изменения в одном сервисе не требуют перезапуска всей системы.
* **Границы протоколов**: Использование REST на границе системы для удобства внешних клиентов и gRPC внутри для максимальной скорости.

### 2. Почему вы выбрали именно этот стек технологий?
* **Python**: Оптимален для быстрой разработки распределенных систем.
* **gRPC**: Выбран из-за бинарного формата обмена данными, что значительно быстрее JSON-over-HTTP.
* **Docker**: Позволяет гарантировать идентичность окружения (Dev/Prod) и автоматизировать сборку gRPC-заглушек.

### 3. Как ваши сервисы справляются с ошибками?
* **Graceful Termination**: Применение `server.wait_for_termination()` в gRPC-сервере позволяет корректно завершать активные запросы.
* **Асинхронность**: Использование очередей сообщений (RabbitMQ) предотвращает потерю данных при временной недоступности сервисов.
* **Логирование**: Внедрено детальное логирование входящих запросов и ошибок для обеспечения наблюдаемости (Observability).

### 4. Как организован деплой и обновление системы без простоя (Zero Downtime)?
* **Rolling Update**: Архитектура подготовлена к обновлению через Kubernetes/Helm, где новые экземпляры контейнеров запускаются параллельно со старыми.
* **Docker Compose**: Локальная оркестрация позволяет тестировать конфигурации сети и зависимостей перед деплоем.

### 5. Что было самым сложным в интеграции всех компонентов?
Самым сложным было организовать автоматическую компиляцию `.proto` файлов внутри Docker-образов и настройку корректного сетевого обнаружения (Service Discovery) между шлюзом и gRPC-сервисом.

### 6. Что бы вы улучшили, если бы у вас был еще месяц?
* **Персистентность**: Переход от хранения в памяти к полноценному использованию PostgreSQL.
* **Мониторинг**: Внедрение Prometheus и Grafana для сбора метрик производительности.
* **Трейсинг**: Добавление Jaeger для отслеживания пути запроса между всеми микросервисами.

## 5. Инструкция по запуску
Для сборки и запуска всех сервисов выполните команду:
```bash
docker-compose up --build
```
