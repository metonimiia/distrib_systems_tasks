# Вопросы для самопроверки

1. Зачем нужен файл `.proto`? Почему мы не пишем классы сообщений сразу на Python?
Файл .proto нужен для описания контракта между клиентом и сервером. В нём задаются структуры сообщений и методы сервиса. На основе этого файла автоматически генерируется код для разных языков программирования. Если писать классы сообщений сразу на Python, не будет единого стандарта взаимодействия, будет сложнее поддерживать совместимость и невозможно автоматически сгенерировать клиента или сервер на другом языке.
2. Что такое Unary RPC? Чем он концептуально похож на REST запрос?
Unary RPC — это самый простой тип взаимодействия в gRPC: клиент отправляет один запрос и получает один ответ. Концептуально он похож на REST-запрос, потому что там тоже есть модель «один HTTP-запрос — один HTTP-ответ». По сути это удалённый вызов функции по сети.
3. Почему gRPC обычно эффективнее REST (JSON) по процессору и трафику?
gRPC обычно эффективнее REST (JSON), потому что использует бинарный формат сериализации Protocol Buffers вместо текстового JSON. Бинарные данные занимают меньше места и быстрее обрабатываются процессором. Также gRPC работает поверх HTTP/2, что позволяет использовать постоянные соединения и мультиплексирование, уменьшая накладные расходы на сеть.
4. Что такое "обратная совместимость" в Protobuf и зачем нужны номера полей (тэги, например `= 1`)?
Обратная совместимость в Protobuf означает возможность изменять структуру сообщений без поломки старых клиентов и серверов. Для этого используются номера полей (тэги), например = 1. В бинарном формате именно номер поля является его идентификатором. Можно добавлять новые поля с новыми номерами, не затрагивая старые, что позволяет обновлять сервис постепенно.
5. Что такое Stub (стаб) на клиенте и Servicer на сервере?
Stub — это сгенерированный клиентский класс, который предоставляет методы для вызова RPC как обычных функций. Он автоматически упаковывает данные, отправляет запрос по сети и возвращает ответ. Servicer — это базовый серверный класс, который генерируется из .proto. Разработчик наследуется от него и реализует бизнес-логику методов сервиса.
6. В каких случаях вы бы выбрали REST, а в каких — gRPC?
REST стоит выбирать, когда нужно сделать публичное API, обеспечить простую интеграцию, поддержку браузеров и удобную отладку через стандартные инструменты. gRPC лучше подходит для взаимодействия между микросервисами, высоконагруженных систем, случаев с жёсткими требованиями к производительности, низкой задержке и строгому контракту между компонентами.
