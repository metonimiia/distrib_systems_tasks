# Ответы на вопросы для самопроверки

## 1. Чем PUT принципиально отличается от PATCH? Какой из них является идемпотентным?

**PUT** — полная замена ресурса. Отправляете весь объект, старый полностью заменяется новым.  
**PATCH** — частичное обновление. Отправляете только поля, которые нужно изменить.

**Идемпотентность:**
- `PUT` — идемпотентен (сколько раз ни отправляй один и тот же запрос — результат одинаков)
- `PATCH` — может быть неидемпотентным (например, операция "увеличить счётчик на 1" при повторном вызове даст разный результат)

## 2. Какие HTTP коды ошибок вы знаете и в каких ситуациях их следует возвращать?

### 400 vs 422
- **400 Bad Request** — общая ошибка: запрос синтаксически некорректен (например, кривой JSON)
- **422 Unprocessable Entity** — синтаксис верен, но семантика нет (например, поле `email` пришло как число, а должно быть строкой, или обязательное поле отсутствует)

### 401 vs 403
- **401 Unauthorized** — пользователь не аутентифицирован (не залогинен, нет токена)
- **403 Forbidden** — пользователь аутентифицирован, но у него нет прав на это действие

### 404
- **404 Not Found** — ресурс не найден (неправильный URL или ID)
### 500
- **500**  - сервер упал.

## 3. Зачем нужны негативные тесты (тесты на ошибки)?

- Проверяют, что API корректно обрабатывает ошибочные ситуации
- Убеждаются, что ошибки возвращаются с правильными кодами и сообщениями
- Защищают от падений сервера при невалидных данных
- Документируют ожидаемое поведение в ошибочных сценариях
- Повышают надёжность и отказоустойчивость системы

## 4. Почему важно возвращать ошибки в структурированном формате (например, JSON), а не просто текстом?

-  клиент может программно обработать ошибку
-  все ошибки выглядят одинаково
-  можно передать код ошибки, поле, вызвавшее проблему, и т.д.
- клиент сам решает, как показать ошибку пользователю
-  легче описать в OpenAPI
