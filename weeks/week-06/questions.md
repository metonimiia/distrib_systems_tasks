# Вопросы для самопроверки

1. Зачем нужны `variables` в GraphQL запросе? Почему нельзя просто склеивать строки?
Переменные (variables) в GraphQL нужны для того, чтобы передавать параметры в запрос отдельно от самого текста запроса. Это делает запросы безопаснее, удобнее для повторного использования и позволяет серверу лучше кэшировать их. Если просто склеивать строки, можно допустить ошибки или уязвимости.
2. Из каких полей состоит стандартный JSON-запрос к GraphQL серверу?
Стандартный JSON-запрос к GraphQL серверу обычно состоит из трёх полей: query, где находится сам запрос, variables, где передаются параметры, и operationName, которое указывает имя операции (если их несколько).
3. Почему у GraphQL обычно только один URL endpoint?
В GraphQL обычно используется один URL endpoint, потому что все действия (получение, изменение данных) описываются внутри запроса, а не через разные адреса, как в REST.
4. Что означает наличие поля `errors` в ответе сервера? Может ли оно прийти вместе с `data`?
Наличие поля errors в ответе означает, что при выполнении запроса произошли ошибки. При этом сервер всё равно может вернуть поле data, если часть запроса была выполнена успешно.
5. Как клиент должен обрабатывать ошибки, если HTTP статус ответа 200?
Если сервер возвращает HTTP статус 200, это не всегда означает успех. Клиент должен дополнительно проверить поле errors в теле ответа и обработать их при необходимости.
6. В чем плюсы использования типизированных клиентов (например, Apollo Client или Relay) перед простыми HTTP-запросами?
Типизированные клиенты, такие как Apollo Client или Relay, удобнее простых HTTP-запросов, потому что они обеспечивают проверку типов, уменьшают количество ошибок, поддерживают кэширование и автоматически обновляют данные на клиенте.
