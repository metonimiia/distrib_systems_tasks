# Вопросы для самопроверки

1. Зачем создавать пользовательскую сеть (bridge network) в Docker Compose, если есть дефолтная?
   Пользовательская сеть дает явный контроль и предсказуемую схему связей между сервисами. Так проще изолировать проект и управлять тем, кто с кем общается.
2. В чем разница между `build` и `image` в описании сервиса?
   `build` собирает образ из Dockerfile локально, `image` берет уже готовый образ из локального кэша или реестра.
3. Что делает `depends_on`? Гарантирует ли он, что приложение в зависимом контейнере полностью инициализировалось?
   `depends_on` задает порядок запуска контейнеров, но сам по себе не гарантирует готовность приложения внутри. Для готовности нужен `condition: service_healthy`.
4. Как работают healthcheck-и и как использовать их в `depends_on`?
   Docker периодически выполняет команду проверки. Если проверки падают, контейнер помечается `unhealthy`; в `depends_on` можно ждать статус `service_healthy`.
5. Как передать переменные окружения в контейнер через Compose (минимум 2 способа)?
   Через `environment:` прямо в `docker-compose.yml` и через `env_file:` с отдельным файлом переменных.
6. Почему Docker Compose редко используют в продакшене (хотя это возможно)?
   Compose удобен для локальной разработки, но для продакшена обычно нужны более сильные инструменты оркестрации: автоскейлинг, rolling update, отказоустойчивость и сложные политики.
