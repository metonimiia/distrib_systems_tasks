# Вопросы для самопроверки

1. Почему мы создаем Deployment, а не сразу Pod?
   Deployment управляет жизненным циклом Pod-ов: поддерживает нужное число реплик и автоматически пересоздает упавшие Pod-ы.
2. В чем разница между Liveness и Readiness пробами? Что произойдет, если провалится каждая из них?
   Liveness показывает, живо ли приложение: при провале контейнер перезапускают. Readiness показывает готовность принимать трафик: при провале Pod временно убирают из Service.
3. Что такое "requests" и "limits" для CPU и памяти? Почему важно их указывать?
   `requests` — гарантированный минимум ресурсов, `limits` — верхняя граница. Это нужно, чтобы планировщик правильно размещал Pod-ы и один контейнер не забирал ресурсы у остальных.
4. Как Service находит свои Pod-ы? (Подсказка: Labels & Selectors).
   Через labels у Pod-ов и selector в Service: Service направляет трафик только на Pod-ы с подходящими метками.
5. Какой тип сервиса (ClusterIP, NodePort, LoadBalancer) используется для внутреннего общения, а какой — для внешнего доступа?
   Для внутреннего общения обычно `ClusterIP`. Для внешнего доступа используют `NodePort` или `LoadBalancer`.
6. Что такое "Rolling Update" в контексте Деплоймента?
   Rolling Update — поэтапное обновление Pod-ов на новую версию без полной остановки сервиса.
