# Вопросы для самопроверки

1. Что такое Server Streaming? Приведите пример использования.
Server Streaming - это когда клиент отправил один запрос, а сервер присылает много ответов подряд. Пример: лента логов или курсы валют в реальном времени.

2. Чем Server Streaming отличается от просто получения большого списка (List) в одном ответе?
Если это один большой список, нужно ждать весь ответ целиком. В стриминге данные приходят по частям сразу, и их можно обрабатывать сразу.

3. Какие метрики производительности существуют (Latency, Throughput, CPU Usage)?
Latency - как быстро приходит ответ. Throughput - сколько запросов в секунду может обработать сервис. CPU Usage - насколько сильно загружен процессор.

4. Почему важно переиспользовать канал (Channel) в gRPC клиенте при замерах производительности?
Потому что создание нового канала каждый раз тратит лишнее время. Тогда мы меряем не скорость метода, а скорость постоянных подключений.

5. В каких сценариях REST может оказаться быстрее или удобнее gRPC?
REST удобнее для простых API и когда нужно быстро проверить запросы через браузер или curl. На простых задачах разница со gRPC может быть маленькой.

6. Что такое "Head-of-Line Blocking" и как HTTP/2 (и gRPC) решает эту проблему?
Head-of-Line Blocking - это когда один медленный запрос тормозит остальные. HTTP/2 решает это тем, что может вести несколько потоков параллельно в одном соединении, и gRPC этим пользуется.
